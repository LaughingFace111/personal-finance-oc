## 1. 系统全景与领域模型 (System Topology & Domain Driven Design)

### 1.1 技术栈全貌

**后端 (Backend)**:
- 框架: FastAPI (Python 3.12)
- ORM: SQLAlchemy 2.0 (async)
- 数据库: MySQL (asyncmy driver) + SQLite (开发/测试)
- 验证: Pydantic v2
- 认证: JWT (python-jose)
- 迁移: Alembic

**前端 (Frontend)**:
- 框架: React 18 + TypeScript
- UI 库: Ant Design 5.x
- 路由: React Router v6
- 状态管理: Zustand
- 图表: @ant-design/charts
- 构建工具: Vite

### 1.2 实体关系 (ER) 简述

User (用户) → 1:N → Book (账本) → 1:N → Accounts/Categories/Tags/Transactions

- User 拥有多个 Book，账本之间数据完全隔离
- Book 下包含：Accounts (账户)、Categories (分类)、Tags (标签)、Transactions (流水)
- Account 可分为 asset (资产)、credit (信用)、liability (负债) 三种类型
- Transaction 关联 Account、Category、Tags，并通过 counterparty_account 记录转账对手方

## 3. 全局财务与数学引擎 (Global Financial Physics)

### 3.1 账户双轨计算法则

**资产账户 (asset)**:
```
current_balance = opening_balance + Σ(in方向的amount) - Σ(out方向的amount)
```
- 账户类型: cash, debit_card, investment, property 等
- 余额随交易实时更新

**信用账户 (credit)**:
```
Available_Credit = credit_limit - debt_amount - frozen_amount
```
- credit_limit: 总额度
- debt_amount: 已使用欠款
- frozen_amount: 分期冻结占用额度

### 3.2 负债平账引擎

给定目标可用额度 T，求 Delta:

**资产盘盈 (增加可用额度)**:
```
Δdebt = (credit_limit - frozen_amount) - T - debt_amount (结果为负)
direction = "in"
```

**资产盘亏 (减少可用额度)**:
```
Δdebt = (credit_limit - frozen_amount) - T - debt_amount (结果为正)
direction = "out"
```

**公式**:
```
Δdebt = (Credit_Limit - Frozen_Amount) - Target_Available - Debt_Amount
```

### 3.3 分期冻结与时间序列算法

**创建分期**:
```
frozen_amount += total_amount
debt_amount 不变
```

**每期执行**:
```
frozen_amount -= principal_per_period
debt_amount += principal_per_period
executed_periods += 1
```

**提前结清**:
```
frozen_amount -= 剩余本金
debt_amount = 0
status = "completed"
```

### 3.4 尾差算法与跨月防错

**除不尽尾差 (如 10000/3)**:
- 首期多扣: first_installment = total_amount - (principal_per_period * (total_periods - 1))
- 末期少扣: 最后一期 = total_amount - Σ(前n-1期)

**跨月日期处理**:
- 使用 dateutil.relativedelta(months=+1) 计算下月日期
- 如果目标日期不存在，自动回退到该月最后一天

## 5. 数据防线与安全约束 (Data Defenses & Constraints)

### 5.1 绝对幂等性机制

**business_key 生成逻辑**:
```python
business_key = f"{book_id}:{external_ref}:{occurred_at.isoformat()}:{amount}"
```

**数据库约束**:
```python
UniqueConstraint("book_id", "business_key", name="uix_transaction_business_key")
```

- 同一账本下 business_key 绝对唯一
- 导入时先查询 business_key，存在则跳过该条记录
- import_hash 用于防重放攻击

### 5.2 软删除规范

**账户删除**:
- 设置 is_deleted = True
- 保留历史交易记录，account_id 关联不变

**前端展示拦截**:
- 查询时自动过滤 is_deleted = True 的账户
- 已删除账户的交易在列表中显示为 "[已删除账户]"
- 账户选择器中不显示已删除账户

### 6.2 调整总额度 vs 调整可用额度 (物理隔离)

**调整总额度 (credit_limit)**:
- 路由: PUT /api/v1/accounts/{id}
- 行为: 仅修改 credit_limit 数值
- 影响: 不触发 frozen_amount 释放，不生成交易流水
- 适用场景: 银行主动调额

**调整可用额度/平账**:
- 路由: POST /api/v1/accounts/{id}/reconcile
- 行为: 修改 debt_amount，生成交易流水
- 影响: 实时更新可用额度，生成调整记录
- 适用场景: 手动平账、差错调整
